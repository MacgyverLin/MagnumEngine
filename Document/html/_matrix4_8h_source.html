12 // Matrix operations are applied on the left. For example, given a matrix M
13 // and a vector V, matrix-times-vector is M*V. That is, V is treated as a
14 // column vector. Some graphics APIs use V*M where V is treated as a row
15 // vector. In this context the "M" matrix is floatly a transpose of the M as
16 // represented in Wild Magic. Similarly, to apply two matrix operations M0
17 // and M1, in that order, you compute M1*M0 so that the transform of a vector
18 // is (M1*M0)*V = M1*(M0*V). Some graphics APIs use M0*M1, but again these
19 // matrices are the transpose of those as represented in Wild Magic. You
20 // must therefore be careful about how you interface the transformation code
21 // with graphics APIS.
22 //
23 // For memory organization it might seem natural to chose float[N][N] for the
24 // matrix storage, but this can be a problem on a platform/console that
25 // chooses to store the data in column-major rather than row-major format.
26 // To avoid potential portability problems, the matrix is stored as float[N*N]
27 // and organized in row-major order. That is, the entry of the matrix in row
28 // r (0 <= r < N) and column c (0 <= c < N) is stored at index i = c+N*r
29 // (0 <= i < N*N).
